tools.

the agent can see freely; it can act only with consent. the registry of both halves is open in tools.ts.

read-act-tools

← back to delta

every tool the agent has is tagged as read or act. read is observation; act is a side effect. they're handled differently on purpose.

read tools — auto-run

read tools are idempotent and side-effect-free. they observe state and return it. the runtime executes them without prompting, because asking the user for permission to look at a tab they already have open would be paranoid theatre.

  • list_tabs — return the titles + urls of every open tab
  • read_active_page — extract the visible text of the active tab, wrapped in an untrusted-content envelope
  • read_tab(tabId) — same, for a specific tab

page text comes back inside a <page-content> tag the system prompt has framed as data — see untrusted-content.

act tools — gated

act tools change the world. they need explicit consent every time, scoped to (origin, tool). you can always allow a pair (e.g. navigate on wikipedia.org); that grant lives in your settings json, alongside the rest of the config.

  • navigate(url) — set the active tab's location
  • open_tab(url) — open a new tab in the background
  • click(selector) — click an element on the active tab (phase 3.1+)
  • type(selector, text) — type into an input on the active tab (phase 3.1+)

see permission gate for how the runtime intercepts these calls.

the registry is open

every tool — name, description, json-schema arguments, side classification, handler — lives in one file: tools.ts. there are no hidden tools. there is no "tools.ts but actually we also do this" file. what you see in the registry is what the agent can call.

// tools.ts
export const tools: ToolDef[] = [
  { name: "list_tabs",        side: "read", schema: {...}, ... },
  { name: "read_active_page", side: "read", schema: {...}, ... },
  { name: "navigate",         side: "act",  schema: {...}, ... },
  { name: "open_tab",         side: "act",  schema: {...}, ... },
  // …
]

verify on huggingface

the agent's system prompt and the json-schema for every tool are mirrored to the delta-agent-prompt dataset on huggingface. you don't have to clone the repo to inspect what the model is told it can do — you can read it on the web, alongside the model card. the mirror is a build artifact; the repo is the source of truth.
